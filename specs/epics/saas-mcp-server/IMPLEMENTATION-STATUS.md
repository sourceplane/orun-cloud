| MCP3 | OAuth 2.1 | ✅ Shipped | **D1 decided: Option A** — vetted public-client allow-list (`OAUTH_PUBLIC_CLIENTS` in `@saas/contracts/auth`: claude-code, claude-web, cursor, vscode, orun-cloud-dev [loopback-only]; exact redirect match + RFC 8252 §7.3 loopback any-port carve-out), NO open DCR. **R5 held: zero new token plane** — an MCP grant IS an OP1 cli-kind session labeled `mcp:<clientId>` via the existing `client_host` column; the authorization code rides `identity.cli_login_grants` as a third `flow='oauth'` (migration 670: code hashed into the existing `cli_code_hash` + `oauth_client_id`/`oauth_redirect_uri`/`oauth_code_challenge` binding columns; same single-use pending→approved→redeemed semantics). identity-worker: `GET /.well-known/oauth-authorization-server` (RFC 8414, raw JSON; issuer = `OAUTH_REDIRECT_BASE_URL`, authorization_endpoint = console `/oauth/authorize`, S256-only, token auth "none"); `POST /v1/auth/oauth2/authorize/complete` (console-internal, actor-authenticated; ~60s single-use code after consent; `oauth.grant.created` security event); `POST /v1/auth/oauth2/token` (RFC 6749: form-encoded in, RAW OAuth JSON out with `Cache-Control: no-store`, §5.2 error codes; authorization_code verifies client/redirect binding + PKCE S256 then mints via OP1 `mintSession`; **code replay ⇒ `invalid_grant` + revocation of the minted refresh family** + `oauth.code.replay_detected` event; refresh_token = the unchanged OP1 rotation incl. reuse-detection/grace). api-edge auth facade: the two public routes + authenticated authorize/complete (rate-limit family `auth`). mcp-worker: `/.well-known/oauth-protected-resource` (RFC 9728) live off the new per-env `OAUTH_AUTHORIZATION_SERVER_URL` var (404 when unset; worker stays binding-free). Console: minimal `/oauth/authorize` consent page (existing login shell; allow-list-validated params; Approve → authorize/complete → redirect with code+state; Deny → `error=access_denied`; never redirects to an unregistered URI) + the existing Sessions & devices list renders MCP grants via the `mcp:` label (revoke path unchanged, test-verified). SDK: `auth.oauthAuthorizeComplete`. Deliberate descopes: no org selector on consent — tokens are workspace-agnostic per OP1 semantics (same as CLI logins); grants visibility = client label in the existing sessions list (no new page); refresh requests are not client_id-bound (the rotating single-use refresh + family revoke is the defense). Tests: 22 identity-worker (metadata shape; allow-list incl. loopback port wildcard; S256-required; code single-use/TTL; PKCE good/bad; replay→family revoke; refresh rotation; console revoke), 6 api-edge facade, 5 mcp-worker (metadata shape/unset-404/405 + quick-check smoke), 9 contracts (allow-list + RFC 8252 matching), 12 console (consent parsing/redirect building/session labels), 1 SDK. | MCP4 | Resources & prompts | ✅ Shipped | All in `packages/mcp` — both transports inherit via `createMcpServer` (no CLI/worker changes). **Resources** (`resources.ts`, design §6, exactly 2 templates, MIME `text/markdown`): `catalog://{workspace}/{entityKey}` — entity overview via the same OV6 exact-filter emulation as `catalog_get_entity` (D2) + `listCatalogDocs(entityRef)` + the overview doc body (byte-capped with the standard truncation marker), rendered as identity/relations/provenance/doc-index/overview markdown; `entityKey` = **base64url(entityRef)** (refs contain `:`/`/` which can't ride one URI path segment; codec exported as `encodeEntityKey`/`decodeEntityKey`). `runs://{workspace}/{project}/{runId}` — run header + git provenance + per-job status table (`getRun` + `listRunJobs`, same as `runs_get`). **Both templates' `list` callback returns an empty list on purpose**: enumerating a whole org's catalog/run history on `resources/list` would blow the client's context budget for no navigational value — agents discover ids via `catalog_search`/`runs_list`, then attach the URI. Resource reads have no `isError` channel, so a new `ResourceReadError`/`toResourceReadError` in `errors.ts` rethrows with the tool-error framing (`<code>: <detail> (requestId: …)` — forbidden/not_found/validation_failed surface in the protocol error message). **Prompts** (`prompts.ts`, 4): `investigate_failed_run` (orient via `runs_list status=failed` when no runId → `runs_get` → `runs_read_logs` with `fromSeq` paging → root-cause report), `access_review` (`access_explain` + `security_events_list` + `audit_search` → provenance table), `usage_review` (`usage_summary` + `quota_check` + `billing_summary` → ≥ 80% flags), `service_snapshot` (`catalog_get_entity` + `catalog_read_doc` + `runs_list` on the entity's sourceProjectId → service brief). Tests: +24 (13 resources, 11 prompts; protocol-level over `InMemoryTransport`), suite now 97. Tool budget untouched (still 19 tools). |

## Notes for later milestones

- State-plane cursors surface as the `createdAt|id` string the endpoints
  accept (matches the console's encoding); opaque cursors (audit, events,
  security-events, webhooks) pass through verbatim; all paginated tools return
  `data.meta.cursor`.
- The SDK does not export its query option types; `packages/mcp` derives them
  via `Parameters<StateClient[...]>` so they stay pinned to real signatures —
  if the SDK exports them later, switch.
- `webhook_deliveries_list` with no `endpoint` argument lists the workspace's
  endpoints (delivery attempts are per-endpoint; no separate endpoints tool —
  budget).
