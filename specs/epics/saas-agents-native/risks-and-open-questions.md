# saas-agents-native — Risks & open questions

## Risks

| # | Risk | Mitigation |
|---|---|---|
| N1 | **SDK dependency risk.** `agents` is a fast-moving vendor SDK; a breaking change or abandonment strands the relay and chat planes. | The blast radius is the shell, by design: attach v1 frames and `RelayCore` are ours (locks 1–2), the chat vocabulary lands in `packages/contracts`, and §12 keeps SDK-native wire formats out of every contract. Shedding the SDK is a re-shell, not a re-design — the same posture that made the AL6→AN1 swap cheap. Pin + renovate like every dep. |
| N2 | **The double-brain problem.** Users conflate the Workspace Agent (chat) with the session agent (sandbox) — "why did the chat say X but the session did Y?" | The product language keeps two nouns visibly distinct: the thread renders child sessions as *cards with their own identity* (never inlined as the agent's own voice); spawns say "started session as_…"; the sealed log remains the only execution truth and the chat links to it rather than paraphrasing verdicts. |
| N3 | **Prompt injection in the chat loop.** MCP results, child-session events, and memory entries are attacker-influenceable content feeding a tool-calling loop. | Structural containment over prompt hygiene: no executing tools, human-only verdicts, gated spawns, visible memory writes, unprivileged worker, owner-credential RBAC on every call (design §8). AN7 ships injection fixtures as CI regression, extending MCP8's eval pattern. |
| N4 | **KV→SQLite DO migration loses live-session continuity.** A session mid-flight during cutover could strand its relay. | Epoch routing: existing sessions stay on the KV class until seal (drain bounded by lease TTL + retention); only new sessions land on `AttachRelay`. The relay is a projection — worst case is a re-attach, never lost truth (§2.4). Staging drain rehearsal is an AN1 done-when. |
| N5 | **Chat token spend surprises tenants.** A chatty loop on the tenant's own Anthropic key burns real money with no ceiling. | Chat spend joins the AF9 envelopes from AN7 (same graceful refusal), `agents.chat_tokens` is visible per thread, and the morning-brief/digest cadence is opt-in-shaped (mutable per thread). The BYO posture also means the tenant's own provider dashboards see it. |
| N6 | **Hibernation vs. the ack bridge.** The relay's in-memory ack waiters (`waitForAck`) die on hibernation-eviction while a head's input POST is in flight. | Already survivable today (the `no_consumer` timeout ack + body `ref` idempotency); AN1 additionally persists pending-ack refs so a woken DO answers late acks correctly. The WS path shrinks the window further (acks ride the socket, no held POST). |
| N7 | **The amendment erodes.** "Conversational orchestration" scope-creeps until the cloud hosts execution after all. | The boundary test is capability, not intent (design §10): CI asserts chat-worker has no control-plane bindings and no execute-shaped tools in its manifest; verdict writes by agent principals are rejected server-side and tested (AN5 done-when). Any new chat capability must pass the §10 test in review. |

## Open questions

| # | Question | Current lean |
|---|---|---|
| NQ1 | **Thread scope: per-user or shared?** Is a chat thread private to its creator or a workspace-visible object teammates can join? | Ship personal-by-default with explicit share (matching session read grants); the presence machinery already supports multi-head threads when shared. Revisit with design. |
| NQ2 | **Which model powers the voice?** Pin the chat loop's model per workspace, or let the tenant configure (they already BYO the key)? | Tenant-configurable from the workspace's provider connection with a sane default; record the model per turn (the AL provenance habit). No gateway abstraction until a second provider is real. |
| NQ3 | **Should the TUI get the chat?** `orun agent attach` gives terminals the *session* head — does the Workspace Agent deserve a terminal surface too? | Not in AN. The thread is a cloud noun (memory, MCP, scheduling live there). If demand shows, it's an api-edge client like every head — nothing in the design blocks it. |
| NQ4 | **Memory beyond facts.** Should memory auto-capture (summarize threads, learn from sealed sessions) rather than only explicit `remember`? | Explicit-only in AN6 — auto-capture without provenance discipline is how hidden state returns. Auto-*suggestions* ("want me to remember this?") are a cheap later increment that keeps the write visible. |
| NQ5 | **AN0 timing.** The WS body binding is a latency win, not a blocker — when does it earn its slot in orun's queue? | After AN1 proves the WS relay in production; the long-poll interval only dominates verdict latency once heads are sub-second (AN2), so sequence it against felt pain, not symmetry. |
