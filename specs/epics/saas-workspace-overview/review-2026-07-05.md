Status: Review — **full-system pass over the shipped epic** (WO1–WO5 merged;

`orun` (`internal/catalogresolve/resolve_full.go`, `internal/objplan/catalog.go`,
`internal/nodes/{assemble,model}.go`, `internal/catalogmodel/entity_envelope.go`)
and `orun-cloud` (`apps/state-worker/src/catalog-projection.ts`,
`handlers/repo-facets.ts`, migrations `460`/`570`,
`web-console-next/src/lib/catalog-portal/page.ts`,

[`saas-catalog-docs`](../saas-catalog-docs/) (cluster **CD**).

## Verdict

The epic **delivered its thesis**: `/orgs/{slug}` is a real front door, the
narrative is git-authored, travels as a content-addressed blob in the catalog
closure, and renders by digest with zero git-provider coupling. Every locked
decision survived contact with implementation — most notably **docs riding the
existing `blob` kind**, which is now paying compound interest: the successor
epic can ship a *multi-doc* system with **no object-plane change at all**.

But the shipped cut is narrower than the model it ratified. `model.md` §2a
promises a docs convention "spanning **every** entity kind" and §6 promises
"Component / System / Domain page — doc blob from R2 by digest"; the code ships
that for exactly **one kind (`Repo`) and one doc (`overview`)**. Meanwhile the
console's entity Docs tab fills the gap with **synthesized markdown presented as
repo files** — the one place the platform's own invariant is violated in spirit.
The findings below are the delta between the ratified model and the built
system; consolidating them is the charter of `saas-catalog-docs`.

## What shipped well (decisions vindicated)

- **`blob` closure, no new object kind** (`architecture-review.md §B`). Zero
  migrations on the object plane, zero CLI↔platform release ordering — and the
  same property now extends to N docs per entity for free. This was the single
  best call in the epic.
- **`repo_facet` keyed by project** (`model.md §2c/§4a`). The repos list and
  the Overview identity join on `source_project_id`; the ref-normalization
  trap was avoided entirely.
- **Kind-agnostic projector.** `org_catalog_entities.kind` as free TEXT meant
  `Repo` landed with no server enum change, exactly as predicted — and any
  future kind (`Product`, CD's enriched kinds) inherits that.
- **Read-edge assembly, no `/overview` endpoint.** The page composes from
  per-context reads; `18-state.md` ownership boundaries stayed intact.
- **Phased delivery worked.** WO2 (the landing) shipped from orun-cloud alone
  (#272) weeks before the cross-repo chain (#434–#436, #276–#278); a
  narrative-chain slip never held the front door hostage.
- **Doc-read authorization resolves digests through the read model.**
  `handleGetOrgCatalogDoc` (`handlers/repo-facets.ts:124`) only serves a digest
  that a projected `doc_ref` in the caller's org actually references — the doc
  endpoint cannot be used as a generic cross-project blob reader.
- **Post-ship reliability hardening landed.** The `ctx.waitUntil` projection
  teardown (silently-frozen console, observed on the ogpic workspace) was
  root-caused and fixed with the `state.catalog_projection` outbox + cron sweep
  (migration `570`) — the projection now converges even when the on-advance
  task dies. Lesson recorded: **a derived read model needs a reconciliation
  loop, not just an event trigger.**

## Findings (as-built vs. the ratified model)

### F1 — The universal docs convention shipped as a Repo-only implementation

`EntityDocs.Overview` exists on every kind
(`orun/internal/catalogmodel/entity_envelope.go:138`), but doc **bytes** are
walked into the closure only for the `Repo` entity: `objplan/catalog.go:112-124`
populates `PendingDocs` solely in `repoEntity()`, while component entities emit
their docs as **bare path strings** via `docsBlock()`
(`objplan/catalog.go:349-365`) — no content, no digest. A component, System, or
Domain with `docs.overview` projects a `doc_ref` of `{path}` only; the console
can never render it. `model.md §6`'s "Component / System / Domain page — doc
blob from R2 by digest" is unimplemented for every kind but `Repo`.

**Disposition → CD1** (walk every emitted entity's docs through the already-
generic `PendingDocs` seam — `nodes/assemble.go:451-477` needs no change).

### F2 — The console fabricates entity docs (invariant violation in spirit)

The service page's Docs tab renders `README.md`, `ARCHITECTURE.md`,
`RUNBOOK.md`, `API.md`, `PROVISIONING.md` **generated from catalog fields** —
`docsFor()` + `genReadme`/`genArch`/`genRunbook`/`genApi`/`genProvision`
(`web-console-next/src/lib/catalog-portal/page.ts:217-248`). These are
presented with file names and file icons as if read from the repo. The platform
that prints *"the console renders what git produced; it never becomes a second
source of it"* is fabricating prose git never produced, on its most
developer-facing surface. (The generation is honest about *data* — every fact
is a real catalog field — but the **file framing** is fiction.)

**Disposition → CD4** (real doc sets replace the synthetic files; anything
computed stays visibly badged "derived — not a repo file", never named `*.md`).

### F3 — `doc_ref` carries no commit; the provenance/staleness surface can't be honest

`model.md` specifies `doc_ref = {path, ref, sha, digest}` where `ref/sha` carry
"the human remote + commit for the view-source link", and `design.md §3` renders
"From `<repo>@<short-sha>`" + "N commits behind". As built, WO3b emits
`{path, sha, digest}` where **`sha` is the sha256 of the doc bytes**
(`resolve_full.go:188-191`), not a commit — and no `ref`/commit is recorded at
all (`objplan/catalog.go:116-118`). The rendered provenance line therefore
cannot point at the commit the head was advanced at, and "N commits behind" has
nothing to diff against. The content-sha also duplicates what `digest` already
guarantees.

**Disposition → CD1** (record `commit` — the resolved head commit — per doc;
keep `sha` for wire compat, deprecated).

### F4 — The pinned-commit read (`model.md §3a`) is not implemented

§3a is a MUST: read doc bytes from the git object at the resolved commit, or
refuse to attach on a dirty tree. The shipped read is working-tree, explicitly
"best-effort" (`resolve_full.go:183-191`), with no dirty-tree refusal at attach
time. The autopush path's clean-default-branch gate
(`cmd/orun/catalog_autosync.go`) makes working-tree == HEAD in the common case,
but `plan --push-catalog` on a dirty tree can still push bytes whose recorded
provenance describes a commit that never contained them — silent drift on the
surface whose pitch is *drift-free*.

**Disposition → CD1** (same milestone as F3: attach-time commit resolution
makes the gate natural to enforce).

### F5 — One doc per entity is a ceiling the design already strained against

`design.md §3` specifies a right-rail **"Pinned docs / Important information"**
card ("the extra `docs` blocks") — it never shipped, because there *are* no
extra docs: the model carries exactly one body (`overview`) per entity, and the
legacy `techdocs/runbooks/adrs` pointers remain bodyless paths. The Q4
size-cap decision ("techdocs tree opt-in, size-capped") also never landed —
today there is **no size cap at all** on the overview read
(`resolve_full.go:188` reads the whole file unconditionally).

**Disposition → CD1/CD5** (the `docs.pages` doc-set model + caps; the Overview
docs card ships against real data).

### F6 — Derived kinds have no declaration site for identity or docs

`System`/`Domain` entities are derived from component `spec.system`/`spec.domain`
strings and carry nothing else — no description, no owner, no docs, and **no
place in `intent.yaml` to say more**. The catalog renders Domain/System pages
that can never be documented. The epic's own model table (§6) lists the Domain
page as a doc surface; there is no path from a repo to it.

**Disposition → CD2** (the `catalog.entities` enrichment block: metadata + docs
merged onto *derived* entities — enrich, never create, so "derived" stays
honest).

### F7 — The epic register drifted

`saas-workspace-overview` — a **shipped** epic — appears in neither
`specs/epics/README.md`'s index table nor `specs/roadmap.md`'s register (nor do
`saas-catalog-portal`/`saas-orun-platform` appear in the roadmap table). The
index files describe themselves as normative registers; a shipped epic that
established a cross-repo wire convention is invisible in both.

**Disposition → fixed in this PR** (WO + CD rows added to
`specs/epics/README.md`; broader roadmap reconciliation left to its owner).

### F8 — Two unconnected "docs" systems now exist in the product

`apps/web-docs` (Docusaurus product documentation) and the catalog's
git-authored entity docs are unrelated systems that both answer to the word
"docs". That is fine — they serve different audiences — but the console is
about to grow a third surface (the CD Docs hub), so the vocabulary needs one
deliberate sentence: **product docs** (about Orun Cloud, authored here) vs.
**catalog docs** (about *your* entities, authored in *your* repos, rendered by
digest). CD's design carries the glossary note.

## The strategic read

The overview epic quietly built **general infrastructure** — a content-addressed
doc pipeline, a digest-gated read endpoint, a sanitizing renderer, a projection
with a reconciliation loop — and then exposed it through the narrowest possible
product aperture (one doc, one kind, one page). The gap between "what the spine
can carry" and "what the product shows" is exactly one epic wide. That epic is
[`saas-catalog-docs`](../saas-catalog-docs/): same invariants, same spine, no
new object kind, no provider coupling — generalized to a **doc set on every
entity kind** and a **browsable Docs surface** in the console.
