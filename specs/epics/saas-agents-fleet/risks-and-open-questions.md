# saas-agents-fleet — Risks & open questions

## Risks

| # | Risk | Mitigation |
|---|---|---|
| F1 | **Fan-out amplifies every existing risk.** One bad dispatch used to burn one sandbox; a tree burns ten, and a routine does it nightly. | The containment is arithmetic, not judgment: depth/width caps at the door, one shared envelope per tree (AF8), the workspace backstop, and `limit.agent_sessions` counting nodes individually. AF9's spawn-storm fixture proves the caps hold against a hostile parent, not just a polite one. |
| F2 | **Ceiling intersection is subtle set math over sealed contracts.** A bug here silently *widens* a child — the one failure class this epic must never have. | The intersection is a pure function in `packages/contracts` with property tests (child ⊆ parent ∀ inputs, fuzzed); the applied ceiling is sealed into both logs so any widening is evident in replay; AF9 fixtures attack it directly. Narrowing bugs fail safe (a too-narrow child errors loudly on a blocked tool). |
| F3 | **The attention fold becomes noise.** If the queue routinely holds ten items, it is a worse inbox, and users stop looking — the failure mode of every notification center. | The fold has five sources, each bounded by design (verdicts are human-gated already; budget marks fire once per envelope; parks are terminal until resumed). No source may be added without a corresponding *removal-by-action* semantics — the no-dismiss rule keeps the queue honest about being a to-do list, not a feed. Digest absorbs everything sub-actionable. |
| F4 | **Track-record gaming.** An agent (or a hostile brief) optimizes the measured rates — small safe PRs to pump merged-clean, self-approving via a second profile. | Rates read sealed logs + external outcomes (IG webhooks, state-plane gates) an agent cannot author; verdict grant rate requires *human* verdicts (agent-answered verdicts are excluded from the record by actor kind); promotion additionally needs the human ack with evidence rendered. The record measures the past; it never *grants* — movement is a human write. |
| F5 | **Routine sprawl.** Dozens of stale routines firing forever, spending quietly. | Per-firing + per-window budgets are mandatory on every routine (no unlimited default); the digest lists every firing with cost; two failures park; a routine unfired-by-predicate for a configurable window surfaces in the digest as "idle — consider disabling". |
| F6 | **Graceful interrupt depends on a cooperative runtime.** A wedged loop ignores the `budget_exhausted` interrupt and keeps spending. | The interrupt is layered over the existing kill chain, not instead of it: no ack within a grace window → lease-refusal on next token refresh (the AG6 kill semantics) → the credential dies within one TTL regardless of cooperation. Cost overshoot is bounded by grace + TTL, and the envelope math records the overshoot. |

## Open questions

| # | Question | Current lean |
|---|---|---|
| F-Q1 | **Which profiles may spawn?** Is `agent.session.spawn` grantable to any profile, or only to types that declare delegation in their sealed contract? | Sealed-contract-declared only (the type says "I orchestrate"), then the grant is the workspace's veto on top. Two locks are better than one on the recursion door. |
| F-Q2 | **Default tree caps.** Depth 2 / width 5-per-parent / 10-per-tree are guesses. | Ship conservative defaults behind workspace config; revisit with AF9 load data. Depth > 2 has no known product story yet — sub-orchestrators are a smell until a real WH case demands them. |
| F-Q3 | **Digest channel + cadence.** Email, Slack (IH), or both; daily or per-quiet-hours-end? | Follow the shipped notification-preference plane (B2): default daily email, Slack when IH's `slack_app` channel is connected. Never more than daily in v1. |
| F-Q4 | **The promotion bar.** Are `≥ 20 sessions · merged-clean ≥ 85% · zero containment` the right workspace defaults, and is the bar per-ladder (dispatch vs action autonomy) or shared? | Per-ladder bars, shared evidence window; the defaults ship as config with the rationale documented. The bar is a product decision that should be tuned against dogfood records before hardening (this repo's own `agents/` loop is the corpus). |
| F-Q5 | **Cross-workspace fleets.** Accounts with many workspaces will want one attention queue and one budget view. | Out of scope here (the scope boundary excludes it); the account-layer seam is WID/TH's Account Hub — the attention fold is deliberately a per-workspace read so a hub fan-out can compose it later without rework. |
| F-Q6 | **Judge verdicts vs approval cards.** Do judge results ever need human countersign before the orchestrator proceeds? | Not structurally — judges inform the orchestrator, humans gate at adoption (WH) and at gated tools (approvals), which cover the consequential moments. If a workspace wants judge countersign, that is an action-autonomy policy on the orchestrator (`ask` on `agent_await` synthesis), not a new mechanism. |
