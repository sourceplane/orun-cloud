# Task 0130 — PERF1: Console client cache, stale-while-revalidate & prefetch

Milestone: `PERF1-console-client-cache`
Branch: `impl/task-0130-console-client-cache`
Spec basis: `specs/roadmap.md` §Performance & Caching (PERF) → PERF1;
`specs/components/12-web-console.md` (Design Direction: skeleton/optimistic).
Owner: `apps/web-console-next` (frontend-only; human-independent — no creds).

## Why (measured 2026-06-02, live stage)

The console feels slow on every navigation because it has **zero client-side
caching**, so it re-pays the (already high) server latency on every page and
shows skeletons each time. Server TTFB: projects ≈ 2.1s, billing/summary ≈ 3.1s
(those are addressed by PERF2/PERF3); this task removes the *perceived* cost by
caching + revalidating on the client.

- `src/lib/use-async.ts` (`useAsync`): no cache, `loading` starts `true`,
  refetches on every mount, no request dedupe.
- `src/lib/use-org.ts` (`useOrgBySlug`, used by `OrgScope` which wraps **every**
  per-org page): calls `client.organizations.list()` on every page mount just to
  resolve one slug → a full org-list round trip per navigation.
- Per-org pages waterfall: `OrgScope` (org list) → page `useAsync` (resource) →
  nested (environments waits on projects) — 2–3 serial round trips before paint.
- `src/app/(app)/layout.tsx` `useRequireAuth`: blank "Loading session…" gate
  before the shell paints, even when a token is already in localStorage.
- No prefetch on hover; no `@tanstack/react-query`/SWR (not in `package.json`).

Highest perceived-speed win; independent of the backend PERF tasks.

## Outcome ("done")

Navigating between visited pages paints **instantly** from cache and revalidates
in the background; duplicate in-flight requests are deduped; the org list is
fetched once per session (not per page); the shell paints immediately (no blank
session gate when a token exists); hovering a nav link/card warms its data. No
regressions; zero console errors.

## Scope (IN)

1. Add a client query cache — `@tanstack/react-query` (recommended) with a
   `QueryClientProvider` in `src/app/providers.tsx`. Defaults: `staleTime` ~30s,
   `gcTime` ~5m, conservative `refetchOnWindowFocus`.
2. Replace bespoke `useAsync` with a query-cache-backed hook keeping the same
   `{data, loading, error, reload}` surface (minimal call-site churn), or
   migrate call sites to `useQuery`. Stable query keys per resource+scope.
3. **Cache the org list**: `useOrgBySlug`/`OrgScope` resolve from one cached
   `organizations.list()` query (shared key); `OrgScope` must not hit the
   network when the org list is already cached.
4. **Prefetch on intent**: a small `PrefetchLink`/`onMouseEnter` hook that
   `queryClient.prefetchQuery(...)` for the destination's primary resource;
   apply to org cards, project cards, and main sidebar links.
5. **Auth-gate paint**: render the shell (sidebar/topbar) immediately; only main
   content waits on session; cached content paints with no blank gate when a
   token is present. Do not regress redirect-to-login.
6. **Optimistic mutations via cache**: convert existing optimistic flows
   (project/environment archive) to `queryClient.setQueryData` + rollback /
   `invalidateQueries`, removing the local list-mirror state where the cache
   now owns it.
7. All wire I/O stays through `@saas/sdk` via `wrap()`; no raw `fetch`.
8. Tests for the cache hook + prefetch helper (pure logic where possible,
   mirroring `tests/web-console-next`); update tests asserting old refetch
   behavior.
9. `ai/reports/task-0130-implementer.md`.

## Out (OUT)

- No backend changes (PERF2/3/4 own those); this task changes *how often* the
  client pays latency + how it paints, not server time.
- No SSR/RSC rewrite; console stays a Workers-hosted SPA. No contract changes.
