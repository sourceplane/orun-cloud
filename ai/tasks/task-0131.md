# Task 0131 — PERF2: Edge bearer-resolution cache

Milestone: `PERF2-edge-bearer-cache`
Branch: `impl/task-0131-edge-bearer-cache`
Spec basis: `specs/roadmap.md` §PERF → PERF2; `specs/components/01-edge-api.md`;
`specs/components/02-identity.md`.
Owner: `apps/api-edge` (+ `apps/identity-worker` for invalidation). Backend;
human-independent.

## Why (measured)

Bearer→actor resolution runs on **every** authenticated request and adds
~0.45s: api-edge `resolve-actor.ts` calls `IDENTITY_WORKER /v1/auth/resolve`
(one worker hop) → identity-worker opens a DB connection and runs **2 sequential
queries** (session-by-token-hash + user-by-id; or api-key-hash + service
principal). Evidence: unauthenticated 401 ≈ 0.25s vs bad-token 401 ≈ 0.7s. This
cost is paid before any business logic on the hot path and is identical on every
request for a stable session/key — i.e. trivially cacheable.

## Outcome ("done")

A repeated authenticated request reuses a cached actor resolution and skips the
identity-worker hop + its 2 DB queries, cutting ~0.4s off the hot path. Logout
(and key revocation) invalidate the cache promptly. Security posture is
preserved: short TTL, fail-closed on cache miss/expiry, never cache a denial as
an allow, and never store secret material (key the cache on a hash of the token,
store only the minimal actor context).

## Scope (IN)

1. In `apps/api-edge/src/resolve-actor.ts`, wrap the identity call with a cache:
   - **Key**: a SHA-256 hash of the bearer token (never the raw token).
   - **Store**: the resolved `ActorContext` only (actorType, actorId, orgId?,
     projectId?, displayName?, email?) — no token, no secrets.
   - **TTL**: short (~30–60s), bounded so revocation latency is acceptable.
     On miss/expiry → resolve via identity-worker as today (fail-closed).
   - **Backend**: Workers **KV** (recommended, survives isolate recycling) or
     the Workers **Cache API** (no new resource). Implementer chooses and
     justifies; if KV, add the binding (see infra note).
2. **Invalidation**: on logout (`/v1/auth/logout`) and on session/key
   revocation, evict the cache entry. Simplest correct approach: have api-edge
   delete the KV/Cache entry on the logout path; document the revocation-latency
   window (≤ TTL) for non-logout revocations.
3. Negative results: do NOT cache 401/denied resolutions (or cache only briefly
   and never as allow). Never let a cached entry widen scope.
4. Tests: cache hit skips the identity call; miss falls through; expiry
   re-resolves; logout evicts; a malformed/denied token never yields a cached
   allow; the hash keying never logs/stores the raw token.
5. If KV is used: `infra/terraform/**` KV namespace + `wrangler.jsonc` binding
   for api-edge (stage + prod), wired through the Orun composition. Keep it
   minimal and env-scoped.
6. `ai/reports/task-0131-implementer.md`.

## Out (OUT)

- No change to identity-worker's resolution *logic* (only an optional logout
  eviction hook). No new auth model. No long TTLs that meaningfully delay
  revocation. No caching of secrets/tokens/raw payloads.

## Open decisions (implementer, justify)

- KV vs Cache API (KV is more reliable across isolates but adds a resource +
  has eventual-consistency write semantics; Cache API is colo-local). TTL value.
  Whether to also cache api-key resolutions (longer-lived) vs sessions only.

## Verification

- Unit tests green; typecheck + lint clean.
- Live stage: measure TTFB of a hot authed endpoint (e.g. `/auth/profile` or
  `/organizations`) before vs after — expect ~0.3–0.45s reduction on cache hit.
  Confirm logout then immediate re-request re-resolves (no stale allow).
