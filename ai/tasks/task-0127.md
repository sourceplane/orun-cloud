## Why this task

The console (`apps/web-console-next`) is already structurally strong: App Router
on Cloudflare Workers, URL-driven 3-level scope switcher (no `sessionStorage`),
Cmd-K palette, designed empty + skeleton states, a four-reason `412` upgrade
insight, dark-mode-by-default CSS-variable theming, and it consumes `@saas/sdk`
(the bespoke `api.ts` was retired). U1–U10 are substantially landed.

A code-grounded audit (2026-06-02) found the remaining gaps are **missing
buyer-facing surfaces that the public API already backs** — i.e. building them
invents no new contract — plus design-system and interaction polish:

1. **No usage / quota view.** `metering.getUsageSummary`, `checkQuota`, and
   `listQuotaViolations` exist on the SDK and `api-edge`, but the only
   usage-adjacent surface is the billing entitlement *limits* table
   (`apps/web-console-next/src/app/(app)/orgs/[orgSlug]/billing/page.tsx`).
   There is no consumption/usage-over-time view and no sidebar entry. Spec 12
   Required Flows: "review usage and quota state."
2. **No notification preferences.** `notifications.getPreferences` (GET) and
   `updatePreferences` (PUT `/v1/notifications/preferences`) exist; there is no
   page or component. Spec 12 Required Flows: "configure ... notification
   preferences."
3. **Account settings is security-only.** `auth.getProfile`, `updateProfile`,
   and `logout` exist; only `/account/security` is built — no profile/general
   page (name/email/sign-out). Spec 12 Required Flows: "manage account."
4. **Org/project surfaces have no settings home.** `orgs/[orgSlug]/page.tsx`
   just `redirect()`s to `/projects`; the config page notes org settings "will
   appear here as more contracts surface." Projects/environments are read-only
   in the UI even though `archive` exists on the SDK.
5. **Design-system primitive gaps.** Spec 12 names a minimum primitive set;
   `Select`, `Sheet`, `Tooltip`, `Popover`, and `Checkbox` are missing from
   `src/components/ui/`. The sidebar is `hidden md:flex` with **no mobile nav**.
6. **Polish gaps.** The Cmd-K command set is hardcoded (not an extensible
   registry); there are **no optimistic mutations** anywhere (every mutation is
   submit → await → `reload()`).

## What "done" looks like

A non-CLI user can, from the console alone and on both desktop and mobile:
review usage and quota state on a dedicated surface; configure notification
preferences; edit their account profile and sign out; reach an org/project
settings home that surfaces metadata and the (API-backed) archive/danger-zone
action; and experience a console whose primitive set, mobile navigation, and
mutation feedback read as "Vercel-standard." Every new surface consumes
`@saas/sdk` only, invents no field absent from the public contracts, and ships
with designed empty + skeleton states.

## Grounding (real code — trust these, verify before editing)

- Console root: `apps/web-console-next/src`. Primitives live in-app at
  `src/components/ui/` (there is **no** `packages/ui`; spec 12 permits a
  "documented in-app design-system directory" — keep them in-app).
- SDK client surface (`packages/sdk/src/`), all already wired through `api-edge`:
  - `metering.ts`: `getUsageSummary(orgId, query)`, `checkQuota(orgId, query)`,
    `listQuotaViolations(orgId, query)` (cursor-paginated).
  - `notifications.ts`: `getPreferences(query)`, `updatePreferences(body)`
    (PUT), `get(...)` (list), `suppressRecipient(...)`.
  - `auth.ts`: `getProfile()`, `updateProfile(...)`, `getSession()`, `logout()`.
  - `projects.ts` / `environments.ts`: only `list`/`get`/`create`/`archive` —
    **no `update`/rename**. `organizations.ts`: only `list`/`get`/`create` —
    **no `update`**. Do not build rename UI; it has no API.
- Console SDK glue: `src/lib/api.ts` (`createClient`, `wrap`, `ApiResult`).
  Every page consumes the SDK via `wrap(() => client.<area>.<method>())` and the
  `useAsync` hook (`src/lib/use-async.ts`). ZERO raw `fetch(` in pages — keep it.
- Shell: `src/components/shell/{sidebar,topbar,scope-switcher,command-palette}.tsx`.
  Add nav entries here; reuse `EmptyState`, `Skeleton`/`SkeletonTable`, `Card`,
  `Table`, `Badge`, `Tabs`, `ZodForm`, `useToast`.
- Theming: `src/styles/globals.css` + `tailwind.config.mjs` are HSL CSS
  variables. New components must use `hsl(var(--token))`; NO hex/rgb literals.
- Existing reference surfaces to mirror for tone/structure: the webhooks detail
  page + its dialogs (`src/components/webhooks/*`), the billing page, and
  `src/components/precondition/insight.tsx`.

## Scope (IN)

This milestone may land as ONE large PR or a short, self-consistent sequence the
implementer manages (suggested split below). Either way it is not done until the
work is on GitHub as open/merged PR(s).

1. **Usage & quota surface** — new route
   `orgs/[orgSlug]/usage/page.tsx` + sidebar entry ("Usage", under the Org
   section). Render a usage summary over `metering.getUsageSummary` (metric +
   bucket selector; reuse a new `Select` primitive) and a quota-violations
   table over `listQuotaViolations` with cursor "Load more" (mirror the audit
   page pagination). Designed empty + skeleton states. Read-only.
2. **Notification preferences** — new route `account/notifications/page.tsx`
   (and an Account-section sidebar entry) over `getPreferences` /
   `updatePreferences`. Per-channel × per-category toggles driven by the
   contract enums (`NotificationChannel`, `NotificationCategory`); save via PUT
   with a toast; optimistic toggle with rollback on error.
3. **Account profile/general** — new route `account/page.tsx` (or
   `account/general`) over `getProfile` / `updateProfile`, plus an explicit
   **Sign out** action (`auth.logout()` + clear token). Make `/account` the
   Account section landing; keep `/account/security` as a sibling tab/route.
4. **Org & project settings homes** —
   - `orgs/[orgSlug]/settings/page.tsx`: read-only org metadata (name, slug,
     id, created) + a **Danger zone** placeholder that is wired only to
     API-backed actions (none exist for org today → render a disabled,
     clearly-labelled "contact support to delete" state; do NOT fake a delete).
   - Give project/environment a settings affordance limited to the **archive**
     action that already exists on the SDK (`projects.archive`,
     `environments.archive`), with a confirm dialog (mirror the webhook
     delete/disable dialogs) and optimistic list removal + rollback.
   - Replace the bare `orgs/[orgSlug]` redirect only if you add a real overview;
     otherwise leave the redirect and add `settings` as a sibling.
5. **Design-system primitives** — add to `src/components/ui/`: `Select`
   (Radix), `Sheet` (Radix Dialog side variant), `Tooltip` (Radix), `Popover`
   (Radix), `Checkbox` (Radix). Token-driven, dark-mode-correct, focus-ring
   consistent with existing primitives. Register the ones you use; do not add
   dead primitives you don't consume in this task (prefer adding `Select`,
   `Sheet`, `Checkbox`, `Tooltip` which the surfaces above need).
6. **Mobile navigation** — replace the `hidden md:flex` dead end: on small
   screens render a hamburger in the topbar that opens the sidebar in a `Sheet`.
   Scope switcher stays reachable on mobile.
7. **Cmd-K extensible registry** — refactor `command-palette.tsx` so commands
   come from a registry/provider that page areas can contribute to (a typed
   `registerCommands(...)` or context array), then register the new
   Usage / Notifications / Account-profile / Settings jumps through it. Behavior
   parity with today's hardcoded set, plus the new entries.
8. **Optimistic mutations where safe** — apply to: notification-preference
   toggles, project/environment archive, and member role change
   (`memberships` role update if exposed) — optimistic update + rollback +
   error toast. Do NOT make creates optimistic.
9. Tests: component/route tests for each new surface (loading, empty, error,
   success, optimistic rollback) following the existing console test pattern;
   primitive smoke tests where the repo tests primitives.
10. NEW `ai/reports/task-0127-implementer.md` (intent, PR boundary, the
    accept/reject of each open decision, verification evidence).

## Hard exclusions (OUT)

- NO new field, route, or workflow absent from `@saas/contracts` / `@saas/sdk`.
  Specifically NO org/project/environment **rename or update** UI (no API), and
  NO faked delete/destructive action that the API doesn't back.
- NO migration / schema / contract change unless a strictly additive *read*
  type is genuinely missing for a listed surface — and if so, justify it and
  keep it byte-stable to existing types; default expectation is **zero**
  contract change.
- NO raw `fetch(` in any page/component — all wire I/O through `@saas/sdk` via
  `wrap()` / `useAsync`, matching the existing convention.
- NO hard-coded color literals; tokens only (`hsl(var(--token))`).
- NO new auth model, OAuth/email provider, or B1 work (human-blocked) — the
  login page stays email-code + bearer-token.
- NO in-app notification **inbox** with read-state (that is P4); this task is
  preferences only.
- NO `packages/ui` extraction (keep primitives in-app), NO secret/token material
  in logs or reports, NO `infra/`, `ai/deferred.md`, or cloudflare provider
  changes. NO lockfile/package.json churn beyond what new Radix primitive deps
  legitimately require (justify any added dependency).
