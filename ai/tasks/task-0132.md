# Task 0132 — PERF3: DB connection reuse & query efficiency

Milestone: `PERF3-db-efficiency`
Branch: `impl/task-0132-db-efficiency`
Spec basis: `specs/roadmap.md` §PERF → PERF3; `specs/components/00-foundation`;
`ai/context/open-risks.md` (N+1 + index notes).
Owner: `packages/db` + the worker handlers that open DB connections. Backend;
human-independent.

## Why (measured + code-confirmed)

The dominant server cost is **per-request DB connection setup repeated in every
worker**. `packages/db/src/hyperdrive/executor.ts` `createSqlExecutor` builds a
fresh `postgres(connectionString, { max: 5, prepare: true })` client per call
and `sql.end()`s it on dispose; every handler does this per request. So each
worker pays a TLS/auth handshake to Hyperdrive→Supabase before its first query,
and that recurs across identity + membership + resource workers per request.
Repeat identical SELECTs never speed up → Hyperdrive's query cache isn't being
realized (short-lived connections; cache state resets). Compounding issues:

- `billing-worker check-entitlement` opens **two** executors per request
  (`repoFactory` + `recorderFactory`).
- `getBillingSummary` (`packages/db/src/billing/repository.ts`) runs **4
  sequential** queries (customer → subscription → plan → entitlements).
- member-list (`membership-worker list-members`) is **N+1**: one
  `listRoleAssignments` per member in the page (open-risks.md).
- reverse-lookup index gap on `membership.organization_members (subject_id, …)`
  for "orgs for subject" (open-risks.md).

## Outcome ("done")

Per-request DB overhead is amortized (no redundant connect/dispose churn); hot
reads are cache-eligible under Hyperdrive and measurably faster on repeat; the
worst offenders (billing summary, member list) drop their round-trip counts.
Target: a single org-scoped read endpoint's server TTFB roughly halves.

## Scope (IN)

1. **Connection reuse**: eliminate the per-request connect/dispose tax where
   safe. Investigate a warm-isolate **module-scoped postgres client** reused
   across requests (Cloudflare keeps isolates warm; Hyperdrive pools
   server-side) vs. the current create-and-`end()` pattern. At minimum, never
   create more than **one executor per request** per worker (fix
   `check-entitlement`'s double executor; share one executor across the repos a
   handler needs). Measure module-scoped reuse; adopt if it's faster and safe
   under Workers I/O constraints, else document why not.
2. **Stop disposing needlessly**: don't `sql.end()` a pooled client mid-isolate
   if it's being reused; only dispose request-scoped clients.
3. **Parallelize `getBillingSummary`**: `Promise.all` the independent reads
   (customer + active subscription; then plan + entitlements), or replace with a
   single JOIN/aggregate query. Keep the public `BillingSummary` shape.
4. **Fix member-list N+1**: add a batched
   `listRoleAssignmentsForSubjects(orgId, subjectIds[])` (one query,
   `subject_id = ANY($1)`), and map roles in memory — replacing the per-member
   loop.
5. **Indexes**: add the missing composite/partial index for "orgs for subject"
   reverse lookups (and any other clearly-missing index the queries imply), via
   a new idempotent migration. Respect the migration manifest/checksum
   conventions.
6. **Hyperdrive cache eligibility**: confirm read paths are NOT wrapped in
   transactions and use parameterized queries so Hyperdrive can cache them;
   document the executor's caching contract in `executor.ts`.
7. Tests: billing-summary parallelization (same result, fewer awaits), batched
   role lookup correctness, executor single-instance-per-request, and any new
   repo method. Migration verifier covers the new index.
8. `ai/reports/task-0132-implementer.md`.

## Out (OUT)

- No bounded-context boundary changes (that's PERF4). No schema redesign beyond
  additive indexes. No change to write-path transactional semantics. No secret
  material in logs.

## Open decisions (implementer, justify)

- Module-scoped client reuse vs per-request (measure both; Workers I/O-object
  lifetime constraints may forbid cross-request reuse — if so, say so and focus
  on single-executor-per-request + Hyperdrive caching). Single-JOIN vs
  `Promise.all` for billing summary.

## Verification

- Unit + migration-verifier tests green; typecheck + lint clean.
- Live stage: re-run the latency probe (`/organizations`,
  `/organizations/:org/projects`, `/billing/summary`) before vs after; expect a
  clear TTFB drop, and (if Hyperdrive caching now works) repeat calls faster
  than the first. Member-list TTFB flat as member count grows (no N+1).
