## Why this task (what the U11 + entitlement-gate work uncovered)

Auditing the entitlement gate during U11 surfaced that **billing has only a
read surface**. The `apps/billing-worker/src/handlers/` set is entirely reads:
`check-entitlement`, `get-customer`, `get-summary`, `list-entitlements`,
`list-invoices`, `list-plans`. There is **no write path**:

- `createBillingCustomer`, `createSubscription`, `changeSubscription`,
  `cancelSubscription` (spec-11 Required Capabilities) are **unimplemented**.
- `packages/db/src/billing/repository.ts` exposes `upsertEntitlement(...)` but
  **nothing in the app ever calls it** — entitlements are never materialized.
- Consequently a brand-new org has zero entitlement rows. PR #209 shipped a
  **stopgap**: `decideEntitlement` returns a hard-coded free-tier default
  (`limit.projects=3`, `limit.environments=3`, `limit.members=5`) on
  `not_found` so the REQUIRED bootstrap flows (create project / environment /
  invite) work at all. That stopgap is a code constant, not a plan — it cannot
  express paid tiers, upgrades, per-org overrides, or a real "free plan" row.

The durable fix is the **provider-neutral subscription + plan-assignment
lifecycle**: assign a plan to an org, materialize that plan's entitlements into
`billing.entitlements`, and update them on change/cancel. This is fully
**human-independent** — spec 11 explicitly allows "manual plans plus a provider
adapter seam if payment integration is staged", so no Stripe credentials are
required. It is the precondition for B6 (Stripe adapter) and U7 (designed
upgrade UX), and it lets the PR-#209 default-map be retired in favor of a real
seeded **free plan**.

## What "done" looks like

- A seeded **free plan** (and at least one paid plan row for catalog/upgrade
  testing) exists with declared entitlement defaults.
- On organization bootstrap, the org is assigned the free plan and its
  entitlements are materialized via `upsertEntitlement` (idempotent).
- `createSubscription` / `changeSubscription` / `cancelSubscription` exist
  (provider-neutral / manual), each re-materializing the org's entitlements from
  the target plan and emitting `subscription.*` + `entitlements.updated` events.
- `check-entitlement` reads real rows for these orgs; the PR-#209 hard-coded
  default-map is **removed** (or reduced to a last-resort safety net with a
  TODO), because every org now has explicit free-plan entitlement rows.
- `getBillingSummary` reflects the active subscription + plan + entitlements.
- The console billing page shows the active plan and real entitlements (no UI
  change required beyond what already reads `getEntitlements`/`getSummary`).

## Scope (IN)

1. Seed plans (migration or idempotent seed): a `free` plan + at least one paid
   plan, each with declared entitlement defaults (projects/environments/members,
   plus a sample `feature.*` flag to prove non-limit entitlements).
2. Plan→entitlement materialization helper that upserts a plan's entitlement set
   for an org (idempotent; `source: "plan"`, carries `subscriptionId`).
3. `createBillingCustomer` + `createSubscription` + `changeSubscription` +
   `cancelSubscription` (provider-neutral), each materializing entitlements and
   emitting `subscription.*` / `entitlements.updated` events.
4. Assign the free plan on org bootstrap (idempotent; safe on retry).
5. Remove the PR-#209 hard-coded default map (or demote it to a documented
   last-resort with a clear TODO) once free-plan rows are guaranteed.
6. `getBillingSummary` returns the active subscription + plan + entitlements.
7. Tests at every layer (db materialization, handlers, bootstrap assignment,
   entitlement read after change), plus an end-to-end check that a fresh org can
   create a project because it has real free-plan rows.
8. `ai/reports/task-0128-implementer.md`.

## Hard exclusions (OUT)

- NO Stripe / real payment provider integration or credentials (that is B6; keep
  a provider-adapter seam but ship manual/provider-neutral). NO secret material.
- NO change to the projects/environments/membership gate *logic*
  (`decideQuantityGate` etc.) — only what billing returns changes.
- NO console redesign — the billing page already reads summary/entitlements;
  only verify it reflects the new real data.
- NO new auth model; NO infra/provider-pin changes beyond a service binding if
  strictly required (justify).

## Open decisions the implementer owns (justify in report)

- Bootstrap assignment mechanism: synchronous membership→billing service-binding
  call vs. an `organization.created` event consumer in billing-worker (weigh the
  race where a user creates a project before async assignment lands — synchronous
  is safer for the bootstrap guarantee).
- Whether to fully delete the PR-#209 default-map or keep it as a safety net.
- Plan catalog shape and default limits (free tier should match or supersede the
  PR-#209 values so behavior doesn't regress).

## Adjacent findings logged during U11 (NOT this task — separate follow-ups)

- **api-edge has no `notifications` facade** (only auth/org/project/webhooks/
  billing/config/metering/audit). The SDK has `notifications.getPreferences/
  updatePreferences`, but the console can't consume them → the U11 notification-
  preferences surface is deferred (`ai/deferred.md`). Needs a `/v1/notifications/*`
  edge facade slice.
- **Console auth-hydration race**: `useRequireAuth` redirects to `/login` before
  `SessionProvider` hydrates the token from `localStorage` on a hard load of a
  deep link (child effects run before the parent's). Real users rarely hit it
  (post-login nav is client-side) but a refresh on a deep link bounces to login.
  Small, high-value fix (gate the redirect on a "hydrated" flag).
- **B1 real auth** (OAuth + email magic link) remains human-blocked on provider
  credentials.
