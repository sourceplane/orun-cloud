# Task 0133 — PERF4: Hot-path hop reduction, parallelization & latency observability

Milestone: `PERF4-hop-reduction-observability`
Branch: `impl/task-0133-perf-observability`
Spec basis: `specs/roadmap.md` §PERF → PERF4; `specs/components/01-edge-api.md`;
`specs/components/03-policy-authorization.md`;
`specs/components/09-events-audit-observability.md`.
Owner: `apps/api-edge` + resource workers (projects/membership/billing/events).
Backend/infra; human-independent.

## Why (code-confirmed)

Hot reads fan out across 3–4 worker hops **serially**. E.g. `GET
…/projects`: api-edge → identity (auth) → projects-worker → membership-worker
(authorization-context, 1 DB query) → policy-worker (in-memory) → projects DB.
Each await is sequential even when independent. `billing/summary` and `audit`
follow the same authorize-then-read serial shape. Policy evaluation itself is
in-memory (cheap) but the **membership-roles fetch + the resource read run
one-after-another** when they could overlap. And today there is **no per-hop /
per-query timing**, so prod latency is invisible and the other PERF tasks can't
be objectively verified.

## Outcome ("done")

Hot read endpoints overlap their independent work (authorization-context fetch ∥
resource read) and shed avoidable hops, reducing wall-clock latency; and every
response carries `Server-Timing` (plus structured timing logs) so per-hop and
per-query cost is measurable in prod and the PERF cluster's gains are provable.

## Scope (IN)

1. **Parallelize authorize ∥ read** on the hot read handlers where the resource
   read does not depend on the authorization result for *what* to read (it only
   gates *whether* to return): start the membership-roles fetch and the resource
   query together (`Promise.all`), then apply the policy decision before
   responding (discard the read on deny). Apply to projects-list, audit-list,
   billing-summary, members-list. Keep deny-by-default semantics intact (never
   leak data on deny).
2. **Reduce hops where safe**: evaluate folding the membership
   authorization-context call into fewer round trips (e.g. a single combined
   "facts + first page" internal call, or co-locating the roles read with the
   resource worker's own DB access) — implementer scopes the smallest safe
   reduction; do NOT violate bounded contexts (no cross-context table reads).
3. **`Server-Timing`**: add a small timing utility (start/mark/measure) threaded
   through api-edge and each worker; emit a `Server-Timing` response header with
   phases (auth, authz, db, total) and structured log lines with the same
   numbers + `requestId`. No secrets/PII in timings.
4. **Metrics sink (optional within this task)**: if cheap, emit the phase
   timings to Analytics Engine (a platform baseline) keyed by route; otherwise
   leave structured logs and note the dashboard as a follow-up.
5. Tests: parallelization preserves deny-by-default (a denied authz still
   returns 403 and never the data), timing util unit tests, header presence.
6. `ai/reports/task-0133-implementer.md`.

## Out (OUT)

- No bounded-context boundary violations (no worker reading another context's
  tables directly). No change to authorization *decisions*, only their timing.
- No new public contract fields. (`Server-Timing` is a transport header.)

## Open decisions (implementer, justify)

- How far to reduce hops without breaking context boundaries (parallelize-only
  vs a combined internal facts+read call). Whether to wire Analytics Engine now
  or defer the dashboard.

## Verification

- Unit tests green (esp. the deny-path-after-parallel-read test); typecheck +
  lint clean.
- Live stage: `Server-Timing` present on responses; the phase breakdown matches
  the external curl TTFB; projects/billing/audit TTFB drops vs the pre-task
  baseline measured in `specs/roadmap.md` §PERF.
