# Objective

Make per-user, per-org email notification preferences reachable end to end:
api-edge facade → console settings page → CLI parity. SDK methods and
notifications-worker handlers already exist.

# Architect Brief

- Bar: **Vercel notification settings** — instant optimistic toggles.
- Security invariant (the one decision that matters): the edge **pins the
  subject to the resolved session actor** — `subjectKind`/`subjectId` from
  the caller are discarded. A user can only read/update their own
  preferences. Org-level defaults stay internal-only.
- Free: page layout, category copy, CLI flag shapes.
- Must not: expose org-subject preferences publicly; add new worker
  handlers; change preference semantics (opt-out model: only explicit
  `false` suppresses).

# Milestone Boundary

`apps/api-edge` (facade + bindings), `apps/web-console-next` (page + nav),
`packages/contracts` (additive internal-actor entry), `packages/cli`
(preferences get/set), and the matching test suites.

# Required Outcomes

1. `GET`/`PUT /v1/notifications/preferences` on api-edge → notifications-worker
   service binding (stage + prod), actor-pinned, rate-limited family.
2. Additive `"api-edge"` entry in `NOTIFICATIONS_INTERNAL_ACTOR_VALUES`.
3. Console `/orgs/:slug/settings/notifications` page: five category toggles,
   optimistic with rollback, opt-out semantics surfaced in copy.
4. CLI `notifications preferences` (get) + `notifications preferences set`.
5. Facade tests proving subject pinning on GET and PUT; pure-model tests for
   the console default/merge semantics.
6. `ai/deferred.md` entry removed.

# Non-Goals

In-app inbox/read state (P4); provider swap (deferred); per-channel beyond
email (V1 contract is email-only).

# Acceptance Criteria

Toggle on stage persists across reload (UI) and reads back via CLI/SDK; CI
green; facade tests prove a forged subjectId is overridden.

# Verification

Suite runs + live stage walkthrough post-merge; prod smoke after promotion.
