## Why this task

Today the only first-party sign-in paths are the email one-time-code flow and
a dev-only bearer-token paste (`apps/web-console-next/src/app/login/page.tsx`).
There is zero OAuth code in the repo. The roadmap (B1) calls for "at least one
OAuth provider (GitHub or Google)". The data model is already OAuth-ready:
`identity.auth_identities (provider, subject)` has a unique index and the repo
already implements `getAuthIdentityByProviderSubject` — nothing calls it yet.

## What "done" looks like

A user can click "Continue with GitHub" on the console login page, complete the
GitHub authorization, and land signed-in at `/orgs` with a normal session
bearer token — using the same session/token model as email login. If a GitHub
account's verified primary email matches an existing user, the GitHub identity
links to that same user (account-linking policy: **link by verified email**).
Otherwise a new user is created. The flow is CSRF-safe (signed `state` +
double-submit cookie) and stores no provider secrets in source control.

## Design

Redirect flow, all through `api-edge` → `identity-worker`:

1. Console navigates the browser to
   `GET {api}/v1/auth/oauth/github/start?return_to={console}/auth/callback`.
2. identity-worker validates `return_to` against an origin allowlist, mints a
   signed `state` (HMAC over `{nonce, provider, return_to, exp}`), sets an
   HttpOnly state cookie holding the nonce, and 302s to GitHub.
3. GitHub redirects to `GET {api}/v1/auth/oauth/github/callback?code&state`.
   identity-worker verifies the state signature + expiry, checks the cookie
   nonce (double-submit CSRF defense), exchanges the code (confidential client,
   client_secret) for an access token, fetches the GitHub user + verified
   primary email, finds-or-links-or-creates the user, issues a session, and
   302s to `{return_to}#token=…&expires_at=…`.
4. Console `/auth/callback` reads the token from the URL fragment, stores it
   (existing localStorage bearer model), scrubs the fragment, and routes to
   `/orgs`.

Provider enablement is config-driven: `GET /v1/auth/oauth/providers` lists only
fully-configured providers so the console renders the button only when GitHub
is wired.

## Human-supplied configuration (NOT committed)

- `GITHUB_OAUTH_CLIENT_ID` (var) and `GITHUB_OAUTH_CLIENT_SECRET` (secret) from
  a GitHub OAuth App per environment (OAuth Apps allow one callback URL each).
- `OAUTH_STATE_SECRET` (secret) — HMAC signing key for the `state` token.
- `OAUTH_REDIRECT_BASE_URL` (var) — public api-edge origin for the env; the
  GitHub callback URL is `{OAUTH_REDIRECT_BASE_URL}/v1/auth/oauth/github/callback`.
- `OAUTH_ALLOWED_CONSOLE_ORIGINS` (var) — comma-separated console origins
  allowed as post-login redirect targets (localhost is allowed by default).

Secrets are applied via `wrangler secret put`; only non-secret vars live in
`wrangler.jsonc`.
