## What Orun Cloud is

Orun Cloud is two things at once, by design:

<div className="signalGrid">
  <article className="signalCard">
    <strong>the managed backend</strong>
    <h3>For orun, the intent compiler</h3>
    <p>
      <a href="https://orun.sourceplane.ai">orun</a> compiles platform intent into
      deterministic plans. Orun Cloud gives those plans a home: managed remote state,
      run coordination across runners, and a service catalog projected from real
      execution history.
    </p>
  </article>
  <article className="signalCard">
    <strong>a SaaS control plane</strong>
    <h3>Tenancy, access, and money</h3>
    <p>
      Identity, organizations, teams, RBAC, configuration, secrets metadata, audit,
      usage metering, quotas, billing, notifications, and webhooks — the primitives
      every production platform needs, shipped as bounded-context services behind
      one edge API.
    </p>
  </article>
  <article className="signalCard">
    <strong>an open baseline</strong>
    <h3>Fork it, rebrand it, grow it</h3>
    <p>
      The whole platform is open source and written as portable component intent,
      converged by orun itself on every commit. <a href="/self-hosting/deploy-your-own">Run
      your own</a> from a clean clone — no committed resource IDs, ever.
    </p>
  </article>
  <article className="signalCard">
    <strong>one design language</strong>
    <h3>Console, docs, CLI, cockpit</h3>
    <p>
      The console, these docs, and orun's CLI cockpit share the same tokens, glyphs,
      and status vocabulary. What you see in a CI log is what you see in the control
      room — and in the margin of this page.
    </p>
  </article>
</div>

## The platform at a glance

```text
                        ┌──────────────────────── api.orun.dev ───────────────────────┐
  orun CLI · SDK ──────▶│  edge API · /v1 · auth resolution · idempotency · rate limits │
  console · CI (OIDC)   └──────┬────────────────────────────────────────────────┬─────┘
                               ▼                                                ▼
             ┌── tenancy & access ──┐  ┌── platform primitives ──┐  ┌── delivery plane ──┐
             │ identity · sessions  │  │ config · feature flags  │  │ remote state       │
             │ workspaces · teams   │  │ audit · events          │  │ run coordination   │
             │ RBAC (deny-default)  │  │ metering · quotas       │  │ catalog of record  │
             │ projects · envs      │  │ billing · notifications │  │ repo links (GitHub)│
             └──────────────────────┘  │ webhooks (signed)       │  └────────────────────┘
                                       └─────────────────────────┘
```

Every box is a bounded-context service with its own contract; nothing reaches the data
plane except through the edge. The [architecture page](/self-hosting/architecture) walks
through each one.

## Why this design

<div className="cardGrid">
  <article className="featureCard">
    <h3>Tenancy in the path, not in headers</h3>
    <p>Every resource lives under <code>/v1/organizations/&#123;id&#125;/…</code>. The tenant is
    visible in every URL, every log line, and every audit entry.</p>
  </article>
  <article className="featureCard">
    <h3>Deny by default</h3>
    <p>No role, no access. Permissions are explicit action strings evaluated by a single
    policy engine, and every decision records how it was derived.</p>
  </article>
  <article className="featureCard">
    <h3>Caller-owned idempotency</h3>
    <p>Unsafe requests accept an <code>Idempotency-Key</code>; replays return the recorded
    response. Retries are safe because you name them, not because we guess.</p>
  </article>
  <article className="featureCard">
    <h3>Converged by its own thesis</h3>
    <p>The platform deploys through <a href="https://orun.sourceplane.ai">orun</a>: every
    commit compiles a plan, every merge converges it. These docs ship the same way.</p>
  </article>
</div>
